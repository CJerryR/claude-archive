# ClaudeArchive 整理手册(交给 Claude Code 执行)

> 目的:把 `ClaudeArchive` 目录里**多个历史版本扩展**产生的、命名/布局各不相同的文件,
> 归类成一套**统一结构**,并安全地去重、隔离垃圾文件。
> 适用目录:`<下载目录>/ClaudeArchive`(本机示例:`C:\Users\17372\Downloads\ClaudeArchive`)。
>
> **执行前务必先整目录做一次备份(整个 ClaudeArchive 复制一份)。** 本手册所有"删除"都先移动到 `_trash/`,不直接删。

---

## 0. 一句话目标

每个对话一个顶层目录;该对话所有产出文件**统一放到 `files/<对话uuid前8>/` 一层目录里**,
**按文件内容去重**:同名文件内容相同只留一份,内容不同的视为不同版本,用 `<原名>__v2.<ext>`、`__v3` 区分;
垃圾/临时文件移到 `_trash/`;历史 JSON 快照保留在 `history/`。

> ⚠️ 重点纠正:早期某版扩展按"轮次"建了一堆文件夹(`files/r12_.../`、`files/r14_.../`…),
> 但 Claude 服务器对同一文件路径**只保留最新版本**,所以那些轮次文件夹里装的多半是**一模一样的重复文件**。
> 本手册要把这些轮次文件夹**按内容去重后合并**进 `files/<对话码>/`,而不是保留它们。

---

## 1. 现状:历史上出现过的全部命名 / 布局规律

### 1.1 四代文件夹布局(可能同时出现在同一对话里)
| 代 | 布局 | 例子 | 处理 |
|----|------|------|------|
| A 最早(≤v1.8) | `files/<文件>` 直接平铺 | `files/CLAUDE.txt`、`files/preview.webp` | 合并进 `files/<对话码>/` |
| B 中期(v1.9–2.1) | `files/<对话uuid前8>/<文件>` | `files/85523dd2/F_hero.webp` | **这就是目标层**,保留 |
| C 过渡(v2.2–2.4) | `files/r<序号>_<时分秒>_<消息uuid8>/<文件>` | `files/r12_131904_019eb513/viewer.html` | 多为重复 → 去重后合并进 `files/<对话码>/` |
| D 现行(v2.5+) | `files/<对话码>/` + 内容版本化命名 | `files/e97526ef/viewer.html`、`viewer__v2.html` | **目标形态**,保留 |

> 对话顶层目录一直是 `<对话名>__<对话uuid前8>`,例:`claudesave1__e97526ef`。
> 目标:A、C 的文件都并入 B/D 所用的 `files/<对话码>/` 单层目录。

### 1.2 同名去重 / 版本后缀(多代混用,需要识别)
| 规律 | 来源 | 例子 | 含义 |
|------|------|------|------|
| ` (N)` 空格括号 | Chrome 下载自动加 | `cjr-writing-SKILL (2).txt` | 纯重复,去掉 |
| `_N` 单下划线数字 | Chrome / 某版扩展 | `preview_3.webp`、`e677c037_1.webp` | 多为重复,靠哈希判 |
| `__N` 双下划线数字 | 老版扩展去重 | `conversation__2.json`、`prompt__3.docx` | 多为重复,靠哈希判 |
| `__<8位十六进制>` | 扩展按文件 uuid 去重 | `p_expanded__a279bc82.webp`、`page-03__2edb75e7.webp` | **版本标识**,不同 hex 多为不同文件 |
| `__vN`(v2.5+ 现行) | 扩展按内容指纹版本化 | `viewer__v2.html`、`claude-archive-suite__v2.zip` | **正式版本号**,保留 |

> 注意:`__<8位hex>` 和 `__vN` 都可能是**有意义的版本区分**,不要当成纯重复无脑删;
> 一切以**内容哈希**为准(见第 3 节):哈希相同才是真重复。

### 1.3 图片
- 一律是 `.webp`(服务器只提供 webp 预览,**原始 PNG 无法获得**,别去找 png)。
- 命名形态:`<13位时间戳>_image.webp`、`<32位md5哈希>.webp`、`<语义名>.webp`、`<uuid>.webp`。

### 1.4 会话级固定文件(保留,不动)
- `conversation.json`、`conversation.md` —— 当前版本的正本。
- `history/conversation_<ISO时间>.json` —— 历史快照(全部保留,查看器用它们自动合并)。
- 根目录 `_index.json` —— 全局索引(保留)。

### 1.5 垃圾 / 异常文件(移到 `_trash/`,不要算进正式归档)
| 模式 | 说明 |
|------|------|
| `未确认 *.crdownload`、`*.crdownload` | 下载中断的临时文件,无效 |
| `~$*.docx`、`~$*.xlsx` | Office 打开时的锁文件,无效 |
| `新建文件夹/`(及任何手动建的副本目录) | 你手动整理时的残留副本,先移走再人工确认 |
| 形如 `<时间戳>_conversation.json`、`conversation__N.json/.md` 出现在 `files/` 里 | 是把会话正本误当附件重复下载的产物,正本已在对话根目录,这些可移走 |

## 2. 目标统一结构

```
ClaudeArchive/
  _index.json
  _trash/                                  # 所有被判定为垃圾/重复的文件,移动到这里(不删除)
    <原相对路径保持不变>/...
  <对话名>__<对话uuid8>/
    conversation.json
    conversation.md
    history/
      conversation_<ISO时间>.json ...
    files/
      <对话uuid前8>/                        # 唯一的产出文件目录(= 对话 uuid 前 8 位)
        viewer.html                        # 某文件的第 1 个(唯一/最早)版本 → 原名
        viewer__v2.html                    # 内容不同的第 2 个版本
        viewer__v3.html                    # …
        claude-archive-suite.zip
        1781..._image.webp                 # 图片、上传文件也都在这一层
        attachment.txt
```

规则:
- **目标只有一层产出目录**:`files/<对话码>/`(`<对话码>` = 对话 uuid 前 8 位,如 `e97526ef`)。
- **布局 B(已是 `files/<对话码>/`)和 D(`files/<对话码>/` + `__vN`)**:已在目标层,保留。
- **布局 A(平铺)和 C(轮次目录 `files/r..._.../`)**:把里面的文件**移动/合并**进 `files/<对话码>/`,
  按第 3 节做内容去重 + 版本化命名;C 的空轮次目录清空后删除(或留空也行)。
- 合并时若与目标目录已有文件**同名**:哈希相同 → 丢弃(算重复,移 `_trash/`);哈希不同 → 作为新版本 `__v2/__v3` 保存。

---

## 3. 去重 + 版本化规则(核心,必须按内容哈希)

对每个**产出文件**(非 conversation.json/.md、非 history、非 _index.json)计算 **SHA-256**,然后:

1. **按 (目标文件名 baseName) 分组**。baseName = 去掉去重后缀后的原始名
   (去掉 ` (N)` / `_N` / `__N` / `__vN` / `__<8hex>`;见第 4 节),保留真实扩展名。
   例:`viewer.html`、`viewer__v2.html`、`viewer__2.html` 的 baseName 都是 `viewer.html`。
2. 在同一对话内,把该 baseName 名下**所有文件按内容哈希去重**:
   - 哈希相同的多个文件 = 真重复 → 只留一个,其余移 `_trash/`(保持原相对路径)。
   - 剩下**几个不同哈希** = 几个真实版本。按时间(文件修改时间/轮次序号)从早到晚排序,
     第 1 个用 `baseName`,其余依次 `baseName__v2.ext`、`__v3.ext`…,统一放进 `files/<对话码>/`。
3. **跨对话不合并、不去重**(不同对话各留各的)。
4. 去重**只认内容哈希**,绝不只看文件名:`p_expanded__a279bc82.webp` 与 `p_expanded__46a16423.webp`
   名字像但哈希不同 = 两个不同文件,都要保留(成为 `p_expanded.webp` 与 `p_expanded__v2.webp`)。
5. **图片/上传文件同样适用**:同一张图被多次下载(不同目录、不同后缀)只留一份;真不同的图各留。

> 实操提示:`files/r..轮次/` 下的 `viewer.html`、`claude-archive-suite.zip` 往往 N 个轮次内容完全相同,
> 去重后通常只剩 1~3 个真实版本 —— 这正是把"一堆 75K/105K 重复文件夹"瘦身的关键。

---

## 4. 文件名归一(求 baseName + 重新版本化)

整理时**先把所有去重/版本后缀剥掉求出 baseName**,再按第 3 节用内容哈希重新分配版本号:
- 剥掉(求 baseName 时):` (N)`、`_N`(纯数字)、`__N`(纯数字)、`__vN`、`__<8位hex>`。
- baseName 示例:`viewer__v2.html` / `viewer__2.html` / `viewer (3).html` → 都归一为 `viewer.html`。

求出每个 baseName 的"几个不同内容版本"后,重新命名:第 1 版用 `baseName`,后续 `baseName__v2.ext`、`__v3.ext`。
**不要保留旧的 `__<hex>` / `__N` 后缀**——统一改用 `__vN`,版本号按内容首次出现的时间顺序。

---

## 5. 执行步骤(建议 Claude Code 按此顺序写脚本,Python 优先)

> 语言随意,Python `pathlib`+`hashlib`+`shutil` 最顺手。**全程 dry-run 先打印计划,确认后再真正移动。**

1. **备份**:确认已整体复制 `ClaudeArchive`。脚本开头打印警告。
2. **扫描**:遍历 `ClaudeArchive/<对话名>__<uuid8>/`(每个对话目录),分类每个文件:
   - conversation 正本 / `history/` / `_index.json` → 跳过不动;
   - 垃圾(第 1.5 节)→ 计划移入 `_trash/`;
   - 其余为产出文件 → 记录 `(对话, 当前相对路径, 布局A/B/C/D, baseName, SHA256, mtime, 大小)`。
3. **垃圾隔离**:把垃圾文件 `shutil.move` 到 `ClaudeArchive/_trash/<原相对路径>`。
4. **按对话 + baseName 分组,内容去重 + 版本化**(第 3、4 节):
   - 目标目录 = 该对话的 `files/<对话uuid前8>/`(没有就创建)。
   - 每个 baseName:同哈希只留一份(其余进 `_trash/`);不同哈希按 mtime 升序命名
     `baseName` / `baseName__v2.ext` / `__v3.ext`,`shutil.move` 进目标目录。
   - 来自布局 A(平铺)、C(轮次目录)的文件都按此并入;B/D 已在目标层,只做去重/重命名。
5. **清理空目录**:被搬空的轮次目录 `files/r..._.../`、以及空的 `files/` 子目录,删除(空目录删除是安全的)。
6. **产出报告**:写 `整理报告.csv`/`.json` 到 `ClaudeArchive/` 根:
   每行 `对话, 原路径, 新路径(或"移入_trash"), 动作(keep/move/version/dedup/trash), 版本号, SHA256`。
7. **结束**:打印统计(对话数、移动数、去重数、版本数、垃圾数、释放空间)。**不要自动清空 `_trash/`**,人工确认后手动删。

## 6. 关键判定:布局/目录正则

```
files 下的子目录名:
  ^r\d+_\d{6}_[0-9a-z]{8}$   # 例 r12_131904_019eb513 → 布局 C(轮次目录,需去重合并)
  ^[0-9a-f]{8}$              # 例 e97526ef           → 布局 B/D(对话码目录 = 目标层)
  # files/ 下直接是文件      → 布局 A(平铺,需合并进对话码目录)
```

对话顶层目录名:`^(?P<name>.+)__(?P<uuid8>[0-9a-f]{8})$`(末尾 8 位十六进制 = 对话 uuid 前 8 位,
即整理目标 `files/<对话码>/` 中的 `<对话码>`)。

---

## 7. 后缀剥离正则(用于求 baseName,第 4 节)

求 baseName 时,把文件名末尾(扩展名之前)的下列任一后缀剥掉(可能叠加,循环剥到没有为止):

```
\s\(\d+\)(?=\.[^.]+$)         # " (2)"
__v\d+(?=\.[^.]+$)            # "__v2"(现行版本号)
__\d+(?=\.[^.]+$)             # "__2"(老版去重)
_\d+(?=\.[^.]+$)              # "_3"
__[0-9a-f]{8}(?=\.[^.]+$)     # "__a279bc82"(老版 uuid 后缀)
```

剥离只为"归一分组",**最终版本号一律重新用 `__vN` 生成**(按内容首次出现时间排序),不保留旧后缀。
注意别误伤正常文件名里的下划线数字(如 `page-01.webp` 的 `-01` 不在扩展名前的 `_\d+` 模式,安全;
但 `fig2_spectrum.webp` 这种中间的 `2_` 不在末尾,也不会被剥)。仅剥**紧贴扩展名之前**的后缀。

---

## 8. 安全红线(务必遵守)

- 绝不 `rm`/直接删除;一切"删除"= 移动到 `_trash/`。
- 绝不动 `conversation.json` / `conversation.md` / `history/` / `_index.json`。
- 跨对话不合并、不去重。
- 改动前先 dry-run 打印完整计划并要求确认。
- `.crdownload`、`~$` 文件可直接判为垃圾(它们对内容无意义)。

---

## 9. 给查看器的兼容性说明(无需脚本处理)

最新版查看器(viewer.html)按"文件名(basename)"在该对话的 `files/` 整个子树里查找文件,
能识别 A/B/C/D 四种历史布局,也认得 `__vN` 版本命名。所以**整理成什么结构都能正常打开**,
本手册的归并主要是"让你眼睛舒服 + 大幅省空间(删掉一堆重复的 75K/105K 副本)",不影响查看器使用。
